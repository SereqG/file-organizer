## Critical

### Unsafe `as unknown as` casts on every ReactFlow custom node

Files:
- `components/nodes/if_node/IfNode.tsx:26`
- `components/nodes/trigger_node/TriggerNode.tsx:21`
- `components/nodes/create_folder_node/CreateFolderNode.tsx:17`

Category: TypeScript / Architecture

Problem:
Every custom node component starts with the same pattern:
```ts
const nodeData = data as unknown as IfNodeData
```
The double `as unknown as` cast tells TypeScript "I know better than you" and erases all type checking on `data` at the entry point of three of the most data-sensitive components in the system. There is no runtime guard either — if the shape mutates (renamed field, missing `config`, etc.), the failure is silent and surfaces as a render-time exception, not a compile error.

Impact:
- ReactFlow node data flows through several places — `WorkflowControls.addNode` (which constructs the `data` object inline), the `useWorkflowEditor` callbacks, and `useWorkflowDefinition` config updates. Any drift between these and the `*NodeData` interfaces is undetectable.
- A future contributor adding a new field to `IfNode['config']` and forgetting to thread it through `WorkflowControls.tsx:49` will get a green build but a broken runtime.
- The same pattern, once accepted, will spread to every new node type.

Recommendation:
ReactFlow exports a generic `NodeProps<NodeT>` (and a `Node<DataT, TypeT>` generic). Type the component signature properly so that `data` is typed at the boundary instead of asserted.

Suggested Refactor:
```ts
// In a shared types file:
import type { Node, NodeProps } from '@xyflow/react'
export type IfRFNode = Node<IfNodeData, 'if'>

// In IfNode.tsx:
export function IfNode({ id, data }: NodeProps<IfRFNode>) {
  // `data` is now typed as IfNodeData — no cast needed
  ...
}
```
Apply the same pattern to `TriggerNode` and `CreateFolderNode`. Register the node-type union with ReactFlow's generics where `<ReactFlow>` is mounted. If the generic route is genuinely blocked (older `@xyflow/react`), at minimum write a `parseIfNodeData(data: unknown): IfNodeData` guard and call it once at the top of each component.

Confidence: High

## High

### Debug `console.log` of workflow state on every render

Files:
- `hooks/useWorkflowDefinition.ts:64`

Category: Code Smell / Performance

Problem:
```ts
console.log('Current workflow definition:', definition)
```
sits in the body of the hook, outside any effect or condition. It fires on every render of every consumer of `useWorkflowDefinition` (and the editor consumes it), printing the full workflow tree each time.

Impact:
- DevTools console gets spammed during normal editing, hiding real warnings.
- Production builds will ship this. On large workflows the serialization cost is non-trivial.
- It is a signal that there is no proper observability hook for state changes — the next contributor will likely add another one rather than remove this.

Recommendation:
Delete the line. If state-change visibility is genuinely useful for development, gate it behind `process.env.NODE_ENV === 'development'` inside a `useEffect(() => { ... }, [definition])` so it logs on changes only, not on every render. Better: install React DevTools and remove this entirely.

Suggested Refactor:
```ts
// Remove the line outright, or:
useEffect(() => {
  if (process.env.NODE_ENV !== 'production') {
    console.debug('[workflow]', definition)
  }
}, [definition])
```

### Suppressed `exhaustive-deps` hides a stale-closure risk in `FolderExplorer`

Files:
- `components/FolderExplorer.tsx:96-99`

Category: React / Business Logic

Problem:
```ts
useEffect(() => {
  startExplore(false)
  return cancelPoll
}, [sessionId]) // eslint-disable-line react-hooks/exhaustive-deps
```
The effect depends on `startExplore` and `cancelPoll` but neither is in the dep array; the lint rule is suppressed with no explanatory comment. Because `startExplore` is itself a `useCallback` depending on `sessionId, cancelPoll, pollJob`, in the current code the closure happens to be stable as long as `sessionId` stays the same. But:
- `pollJob` references `cancelPoll` (which is stable), so that chain holds today by accident, not by design.
- The suppression communicates "I gave up on the rule" rather than "this is intentional, here's why".
- Any future change to `startExplore` dependencies (e.g., reading a prop) reintroduces a real stale-closure bug, and the lint rule is no longer there to catch it.

Impact:
- Silent regressions: a contributor adds a configurable polling interval prop, threads it through `pollJob`, and the next `sessionId` change will still call the stale `startExplore`.
- The cleanup return is `cancelPoll`, which is fine, but it depends on a closed-over `timeoutRef` whose contract isn't enforced anywhere.

Recommendation:
Either rewrite the effect so the deps are actually correct and the lint rule passes naturally, or keep the suppression but document *why*. Restructuring so the polling lives entirely in a ref-driven loop (no `useCallback` chain) is cleaner.

Suggested Refactor:
Extract polling into a dedicated hook `useExploreJob(sessionId)` returning `{ state, retry }`. The hook owns the timeout ref, the abort logic, and the effect — the component just reads `state`. This eliminates the cascading `useCallback` chain that forced the suppression in the first place.

Confidence: High

### `JSON.parse` on drag-drop payload with no error handling

Files:
- `components/WorkflowControls.tsx:56-62`

Category: Business Logic / Security

Problem:
```ts
useEffect(() => {
  dropHandlerRef.current = (event: React.DragEvent<HTMLDivElement>) => {
    const raw = event.dataTransfer.getData('application/node')
    if (!raw) return
    const entry = JSON.parse(raw) as NodeDescriptor
    addNode(entry, { x: event.clientX, y: event.clientY })
  }
}, [addNode, dropHandlerRef])
```
`JSON.parse` will throw on any malformed input. The cast `as NodeDescriptor` then assumes the parsed object matches the descriptor shape — neither is verified.

Impact:
- A drop event with bad payload (browser quirk, a malicious page that drops into the editor iframe, a stale drag from another app holding `application/node`) takes down the canvas with an uncaught exception.
- Even with valid JSON, an arbitrary object reaches `addNode` and ends up as ReactFlow node `data` — which the Critical finding above then accesses without type guards. The two issues compound.

Recommendation:
Wrap `JSON.parse` in a `try/catch`. Validate the parsed shape before passing to `addNode`. A lightweight runtime guard is enough — Zod is overkill if you don't already use it.

Suggested Refactor:
```ts
function parseNodeDescriptor(raw: string): NodeDescriptor | null {
  try {
    const parsed: unknown = JSON.parse(raw)
    if (
      parsed && typeof parsed === 'object' &&
      'kind' in parsed && 'nodeType' in parsed && 'label' in parsed
    ) {
      return parsed as NodeDescriptor
    }
  } catch { /* swallow malformed JSON */ }
  return null
}

// usage:
const entry = parseNodeDescriptor(raw)
if (!entry) return
addNode(entry, { x: event.clientX, y: event.clientY })
```

### Real bug in `walkCondition` missing-field handling — `baseMatched` is always `false`

Files:
- `lib/workflow/evaluator/explain.ts:87-94`

Category: Business Logic

Problem:
```ts
if (!lookup.found) {
  if (strategy === 'error') throw new MissingFieldError(condition.field)
  const baseMatched = strategy === 'skip' ? false : false
  const final = condition.negate ? !baseMatched : baseMatched
  if (final) matched.push(label)
  else failed.push({ condition: label, expected: condition.value, actual: '(missing)' })
  return final
}
```
The ternary `strategy === 'skip' ? false : false` is a no-op — both arms return `false`. So:
- `strategy === 'false'` returns `final = condition.negate`. Correct: missing field counts as a non-match, negation flips it. ✅
- `strategy === 'skip'` should *skip* the condition (treat it as a no-op / passthrough), but here it produces the exact same result as `false` and *still pushes to matched/failed*. ❌

Compare to `evaluate.ts`, where `'skip'` short-circuits the parent group via a different control-flow path. In `explain.ts`, `'skip'` is effectively dead — the trace will record missing-field conditions as failures even when the user asked to skip them.

Impact:
- The `EvaluationPreview` UI (which uses `evaluateWithTrace`) will show "Failed" entries for fields the user explicitly chose to skip — confusing and misleading.
- The boolean routing result of preview vs. runtime evaluation can diverge for `strategy: 'skip'`, breaking the "preview matches execution" contract that makes the evaluator trustworthy.

Recommendation:
Decide the intended semantics for `'skip'` in trace mode (almost certainly: don't push to either list, return a neutral value, and let the parent group ignore it the same way `evaluate.ts` does). Mirror the logic from `evaluate.ts` exactly so preview = runtime.

Suggested Refactor:
At minimum, simplify and fix the dead branch:
```ts
if (!lookup.found) {
  if (strategy === 'error') throw new MissingFieldError(condition.field)
  if (strategy === 'skip') {
    // Match evaluate.ts: skip means "this condition doesn't participate"
    matched.push(`${label} (skipped — missing field)`)
    return condition.negate // or whatever evaluate.ts returns for skip
  }
  // strategy === 'false'
  const final = condition.negate // missing -> false, negate flips
  if (final) matched.push(label)
  else failed.push({ condition: label, expected: condition.value, actual: '(missing)' })
  return final
}
```
The exact final shape depends on what `evaluate.ts` does for `'skip'` — align the two before shipping.

Confidence: High

### Tree components have no keyboard navigation or ARIA roles

Files:
- `components/FileTree.tsx`
- `components/FileTreeNodeItem.tsx`
- `components/nodes/create_folder_node/FolderPicker.tsx`

Category: Accessibility

Problem:
Tree expansion/collapse and folder selection are wired purely to `onClick`. There is no `role="tree"` / `role="treeitem"`, no `aria-expanded`, no arrow-key navigation, no Enter/Space toggling, and no roving `tabIndex`. The `FolderPicker` does at least set `tabIndex={hasChildren ? 0 : -1}`, which is a partial fix, but it does not implement the WAI-ARIA Tree View keyboard contract (Up/Down to move, Right/Left to expand/collapse, Home/End).

Impact:
- Keyboard-only and screen-reader users cannot select a parent folder in `CreateFolderConfigModal` — they cannot complete the workflow.
- The exploration result tree (`FileTree`) is also unreachable without a mouse, which makes the entire "next step" gate inaccessible.

Recommendation:
Implement the WAI-ARIA Tree View pattern (https://www.w3.org/WAI/ARIA/apg/patterns/treeview/). Practically:
- Container: `role="tree"`, owns focus via roving `tabIndex`.
- Each item: `role="treeitem"`, `aria-expanded={open}` when it has children, `aria-selected` when it's the chosen folder (in `FolderPicker`).
- A `useTreeKeyboardNav` hook centralizes arrow/Enter/Space handling so both `FileTree` and `FolderPicker` reuse it.

Suggested Refactor:
Extract a `Tree` primitive in `components/shared/Tree/` that takes a `FileTreeNode` and renders it accessibly. `FileTree` becomes a thin read-only wrapper; `FolderPicker` adds selection on top. This also kills the duplicated `collectDirectoryIds(root, ids, 2)` pattern (see Low finding).

Confidence: High

### Recursive condition editor lacks `React.memo` — deep trees re-render in full

Files:
- `components/nodes/if_node/ConditionBuilder/ConditionGroupEditor.tsx`
- `components/nodes/if_node/ConditionBuilder/ConditionRow.tsx`

Category: Performance

Problem:
`ConditionGroupEditor` is recursive (depth up to 10 per `validateIfConfig`). It renders `ConditionRow`s and nested `ConditionGroupEditor`s. None of them are memoized. Every keystroke in any input bubbles up to the modal's root state, which re-renders the entire tree from the top. With ~10 conditions at depth 3 you already see hundreds of unnecessary renders per keystroke.

Impact:
- Editor feels sluggish on moderate condition trees, particularly on low-end devices.
- The cost grows with workflow complexity, so the issue gets worse over time exactly where it matters most.

Recommendation:
- Wrap `ConditionRow` in `React.memo` with a custom comparator that checks `condition` reference equality.
- Wrap the inner `ConditionGroupEditor` in `React.memo`.
- Ensure callback props are stable (`useCallback` in the parent, with the right deps).

Suggested Refactor:
```ts
export const ConditionRow = memo(function ConditionRow(props: ConditionRowProps) { ... })
```
Pair with `useCallback` in the parent so prop identity is stable across renders. The validation `useMemo` already runs on every change — that's fine because the tree itself is the input.

Confidence: Medium

### Index-as-key in evaluation result lists

Files:
- `components/nodes/if_node/ConditionBuilder/EvaluationPreview.tsx:108`
- `components/nodes/if_node/ConditionBuilder/EvaluationPreview.tsx:117`

Category: React

Problem:
```tsx
{result.matchedConditions.map((c, i) => <li key={i}>{c}</li>)}
...
{result.failedConditions.map((c, i) => (<li key={i}>...</li>))}
```
Both result lists use array index as the React key. The order and contents of matched/failed conditions change every time the user edits the condition group and re-runs the preview, which means React reconciles by position rather than by identity. Combined with the textarea state nearby, this can cause weird focus/animation glitches.

Recommendation:
Use the condition label itself as key (`key={c}` for matched, `key={c.condition}` for failed). Labels are derived deterministically by `describeCondition`, so they are unique within a single evaluation.

### Non-null assertion after a truthiness check

Files:
- `components/nodes/create_folder_node/CreateFolderNode.tsx:30,52`
- `components/nodes/if_node/IfNode.tsx:39-62`

Category: TypeScript

Problem:
```tsx
const configured = !!(nodeData.config?.folderName && nodeData.config?.parentFolderId)
...
{configured ? nodeData.config!.folderName : 'Not configured'}
```
The `!` non-null assertion is needed because TypeScript can't prove that `nodeData.config` is defined from the value of the local `configured` boolean. The pattern works, but it's defensive code that hides a missing type narrowing.

- Stylistic noise that conditions readers to ignore `!` — a future genuine non-null assertion that *should* be a type guard will slip through review.

```tsx
const folderName = nodeData.config?.folderName
const ready = !!(folderName && nodeData.config?.parentFolderId)
...
<div className={...}>{ready ? folderName : 'Not configured'}</div>
```
Or, if you keep the `configured` flag, structure it as a type-guard function: `function isConfigured(d: CreateFolderNodeData): d is CreateFolderNodeData & { config: Required<...> }`.

### User-edited JSON parsed and cast to `WorkflowItem` with no schema check

Files:
- `components/nodes/if_node/ConditionBuilder/EvaluationPreview.tsx:35-44`

Category: Business Logic

Problem:
The evaluation preview lets the user edit a JSON textarea, then:
```ts
item = JSON.parse(json) as WorkflowItem
```
The `JSON.parse` is correctly guarded by `try/catch`, but the cast is a lie — any object shape can be parsed. The downstream evaluator then accesses `item.stat.size`, `item.flags.hidden`, `item.children`, etc. Missing or wrong-type fields will throw deep inside the evaluator, with error messages that point at evaluator internals, not at the user's bad JSON.

Impact:
- Confusing developer experience: the preview is meant to be a sandbox where users learn how their conditions behave, and the error messages they get are unrelated to what they did wrong.
- Trust in the preview erodes — exactly the opposite of what the feature is for.

Recommendation:
Validate the parsed object against the `WorkflowItem` shape before evaluation. If the project doesn't already use a validation library, a hand-rolled `validateWorkflowItem(value: unknown): { ok: true; item: WorkflowItem } | { ok: false; error: string }` is cheap. The errors it returns become user-facing messages in `parseError`.

Suggested Refactor:
```ts
const parsed = safeJsonParse(json)
if (!parsed.ok) { setParseError(parsed.error); setResult(null); return }
const validated = validateWorkflowItem(parsed.value)
if (!validated.ok) { setParseError(validated.error); setResult(null); return }
const trace = await evaluateWithTrace(validated.item, conditions, { missingFieldStrategy: strategy })
```

### Hard-coded 1.5s polling with no backoff and no max attempts

Files:
- `components/FolderExplorer.tsx:64,93`

Category: Performance / UX

Problem:
```ts
timeoutRef.current = setTimeout(() => pollJob(jobId), 1500)
```
The polling cadence is fixed at 1500ms regardless of job state, history, or duration. There is no backoff for long-running scans, no maximum attempt count, and no user-visible "still scanning, X seconds elapsed" feedback. The backend can also return `IN_PROGRESS` indefinitely without the UI ever giving up.

Impact:
- Large directories that legitimately take 30+ seconds hammer the backend with 20+ identical poll requests.
- A backend that hangs (job stuck in `IN_PROGRESS`) leaves the UI in `loading` forever with no recovery.
- Constant requests at 1.5s intervals defeat any HTTP-level caching.

Recommendation:
- Exponential backoff capped at, say, 5s: 1500ms → 2250ms → 3000ms → 5000ms.
- A "Still scanning…" message after N seconds, with a Cancel button.
- A hard timeout (e.g., 2 minutes) that transitions to the `error` phase with a clear message.

Suggested Refactor:
Combine this with the dedicated `useExploreJob` hook proposed in the High finding about stale closures — that hook becomes the natural home for cadence, backoff, and timeout logic.

Confidence: Medium

### Each modal re-implements its own backdrop/close behavior

Files:
- `components/TriggerSelectModal.tsx`
- `components/DepthConfirmModal.tsx`
- `components/nodes/if_node/IfConfigModal.tsx`
- `components/nodes/create_folder_node/CreateFolderConfigModal.tsx`

- `role="dialog"` and `aria-modal="true"`,
- Escape-to-close,
- focus trap + focus return,
- consistent styling (one prop variant if you need a "danger" vs "default" look).

All four current modals become thin wrappers that pass children.

## Positive findings
